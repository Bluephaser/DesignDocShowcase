# DesignDocShowcase
Something that has become rather important to me over the years when writing code for game projects is that any code I write meant to be designer-facing should cause as little friction in the development process as physically possible. That is, when my code is meant to be interacted with by non-programmer members of the team, I try to imagine a worst-case scenario where those members desperately need to use my code but have no way of contacting me. This is why, in addition to working extra hard to make designer-facing code as easy to use as possible, I tend to make detailed documentation to be used by the designers that explains to them how the code works, and how to integrate it into their own work.
